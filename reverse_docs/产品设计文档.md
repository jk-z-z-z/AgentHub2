# AgentHub 产品设计文档

## 1. 产品定位

AgentHub 是一个面向软件交付场景的多智能体协作工作台。它把群组会话、智能体成员、共享代码空间、任务编排、预览、部署和执行记录放进同一个项目上下文中。

产品核心不是单点 AI 问答，而是把 AI 作为团队成员纳入真实协作流程。

## 2. 目标用户

主要面向三类用户：

1. 个人开发者
   需要用智能体辅助写代码、改页面、预览和部署。
2. 小型研发团队
   需要在共享项目空间中协同使用多个智能体。
3. AI 应用搭建者
   需要配置智能体模板、工具、技能和 MCP 能力。

## 3. 核心对象

### 用户

系统登录主体，拥有基础资料和用户画像。

### 群组

系统的一级协作容器，分为：

- `personal`
- `project`
- `bootstrap`

其中 `project` 群是产品主战场，承载代码、任务和交付能力。

### 成员

群组中的参与者，统一抽象为成员：

- `user`
- `agent`
- `system`

### 智能体实例

真正参与执行的运行实体，可挂载工具、技能和 MCP。

### 智能体模板

智能体的标准化原型，定义人格、角色、工具和技能配置。

### Workspace

项目群对应的共享工作区，是执行、预览、部署的基础。

### Task Run / Task Node

任务的结构化表示。Run 表示一次任务过程，Node 表示其中的具体执行单元。

## 4. 产品结构

前端主结构分为五个域：

1. 消息
2. 通讯录
3. 智能体
4. 文件
5. 个人信息

对应的产品含义分别是：

1. 协作入口
2. 组织关系
3. 能力配置
4. 项目资产
5. 用户画像

其中消息页是主工作台。

## 5. 核心能力

## 5.1 会话协作

支持：

- 会话列表
- 群聊消息
- 提及智能体
- 消息事件查看
- 代码 diff 查看

消息不仅是聊天入口，也是系统调度入口。

## 5.2 群组与成员管理

支持：

- 创建群组
- 添加用户成员
- 添加智能体成员
- 删除成员
- 删除群组

群组是项目、任务和交付的边界。

## 5.3 智能体中心

支持：

- 智能体实例创建与管理
- 模板创建与管理
- 全局工具查看
- 全局 MCP 管理
- 全局技能池查看
- 实例级文件、技能、工具、MCP 配置

## 5.4 项目代码工作台

支持：

- 项目代码树浏览
- 文件编辑
- 新建文件和目录
- 删除文件和目录
- 草稿缓存

代码工作台和消息工作台是双入口。

## 5.5 任务规划

支持：

- 创建任务 Run
- 用 DAG 表示任务结构
- 节点认领
- 节点完成
- 节点复核
- 任务图修改

这让系统具备了工程化任务分解能力。

## 5.6 交付闭环

支持：

- 沙箱命令执行
- 预览地址生成
- Docker 部署
- 执行日志记录

系统目标不是只给建议，而是推动真实交付结果产出。

## 6. 典型用户路径

### 个人助理路径

1. 用户进入个人群
2. 发送消息
3. 智能体自动响应

### 项目协作路径

1. 用户创建项目群
2. 添加用户和智能体成员
3. 在群里发起需求
4. 通过管家或智能体推进执行
5. 在文件、任务、预览、部署面板中查看结果

### 智能体配置路径

1. 创建模板
2. 基于模板创建实例
3. 配置文件、技能、工具、MCP
4. 将实例加入群组

## 7. 产品特征

AgentHub 当前有四个明显特征：

1. 以群组而不是单对话作为协作边界
2. 以任务图而不是纯聊天作为复杂任务载体
3. 以智能体配置中心而不是单个 prompt 作为能力入口
4. 以预览、部署和执行结果作为交付闭环

## 8. 当前短板

1. 权限模型较轻
2. 组织层级较浅
3. 任务与消息联动还可以更强
4. 部署能力仍偏本地化
5. 缺少统一运行指标和运营视角

## 9. 建议方向

短期建议：

1. 强化消息、任务、代码三者联动
2. 增强智能体执行可解释性
3. 优化失败反馈与结果展示

中期建议：

1. 增加团队空间和项目模板
2. 增强权限体系
3. 支持多环境部署

## 10. 总结

AgentHub 当前已经形成了清晰的产品骨架：以项目群为中心，组织人类成员、智能体成员、共享代码、任务 DAG 和交付流程。它的核心价值不在“会聊天”，而在“能把协作组织成结果”。
