# AgentHub 单智能体设计

## 1. 单 Agent 的定义

在 AgentHub 里，一个单独的 agent 不是一段 prompt，而是一个完整的数字员工实例，由五部分组成：

1. 实例身份
2. 文件化人格与配置
3. 能力装配
4. 执行引擎
5. 运行上下文

## 2. 核心对象

### AgentInstance

运行时主体，决定：

- 名称
- 描述
- 引擎类型
- 引擎配置
- 状态

### AgentProfile

实例模板，负责提供标准化角色原型，包括：

- `SOUL.md`
- `PROFILE.md`
- `BOOTSTRAP.md`
- tools 配置
- skills 配置
- 模型参数

### Agent 工作目录

实例创建后会有自己的目录，保存人格文件、能力配置和附加知识。

## 3. 上下文设计

单 agent 当前主要运行在两种场景：

1. personal
2. project

### Personal 场景

系统提示词会组合：

- Agent `SOUL.md`
- Agent `PROFILE.md`
- User `PROFILE.md`
- User `MEMORY.md`

### Project 场景

系统提示词会组合：

- Agent `SOUL.md`
- Agent `PROFILE.md`
- Project `PROFILE.md`
- Project `MEMORY.md`

### 短期记忆

短期记忆由外部调用方整理后注入，不由 agent 自己读取消息系统。

### Extra Context

通常包含：

- `user_id`
- `project_id`
- `group_id`
- `run_id`
- `node_id`
- `input_text`

## 4. 构建过程

单 agent 的统一入口是 `invoke_agent()`。

核心流程：

1. 规范化短期记忆
2. 建立 trace
3. 构造工具执行器
4. 构建完整 agent
5. 选择引擎
6. 执行并记录结果

`build_complete_agent()` 负责真正的装配：

1. 读取 `AgentInstance`
2. 判断 personal 或 project 场景
3. 构建系统提示词
4. 加载 toolkit
5. 生成 `EngineContext`

## 5. 能力装配

### Tools

工具由内置定义和实例级开关共同决定。当前工具大致分成：

1. workspace
2. project
3. runtime
4. profile

### Skills

技能用于提供工作方法和行为规则，由 agent 自己的 skill loader 装配。

### MCP

MCP 为 agent 提供外部接入能力，并且可以按实例启停。

## 6. 执行引擎

当前支持：

1. `internal_llm`
2. `agentscope_react`
3. `acp:codex`
4. `acp:claude_code`

这让 agent 角色和执行器解耦，同一个角色可以切换不同后端。

## 7. 工具调用设计

每个内置工具都会被包装成统一工具对象，负责：

- 参数 schema
- 权限透传
- trace 记录
- 错误封装

工具调用最终走：

- 注入的 `tool_executor`
  或
- 系统内置执行器

## 8. 文件系统角色

文件系统在单 agent 设计里很重要，承担三类职责：

1. 人格与配置存储
2. 执行上下文桥接
3. 知识文件承载

这意味着当前 agent 是明显的文件驱动型实例。

## 9. 场景差异

### 个人群中

更像个人数字助理，依赖用户画像和用户记忆。

### 项目群中

更像项目成员，依赖项目画像和项目记忆。

### 任务节点中

更像执行工种，围绕节点目标和执行上下文工作。

## 10. 当前设计特点

### 优点

1. 边界清晰
2. 配置化程度高
3. 易于模板化复用
4. 引擎可替换
5. 运行过程可追踪

### 限制

1. 文件系统耦合较重
2. 提示词仍偏拼接式
3. 能力治理还不够细
4. 长期记忆还不够丰富

## 11. 建议方向

1. 把人格配置逐步结构化
2. 强化工具与能力治理
3. 增加任务级和反馈级记忆
4. 提升工作目录和工作区抽象

## 12. 总结

AgentHub 中的单个 agent，本质上是一个“文件驱动、能力可装配、上下文可切换、执行引擎可替换”的数字员工实例。它已经不是简单的 prompt runner，而是一个可被配置、复用和追踪的执行单元。
