# 前端 UI 与组件规范

## 1. 总则

- 页面只做“路由态 + 数据编排 + 事件转发”。
- 复杂 UI 块必须下沉到 `src/components/`。
- 能用 Element Plus 的地方优先用 Element Plus，不手写同义功能。
- 样式目标是“统一、可复用、可修改”，不是“每页都单独设计一套外观”。

## 2. 目录边界

### `src/pages/`

- 只放路由页。
- 负责：
  - 读取路由参数与 query
  - 调用 `api/`
  - 维护页面级状态
  - 把数据和事件传给组件
- 不放：
  - 业务请求函数
  - 可复用 UI 块
  - 大段手写布局骨架

### `src/components/`

- 放页面内可复用的 UI 块。
- 建议按功能分目录，例如：
  - `messages/`
  - `contacts/`
  - `profile/`
  - `admin-users/`
  - `agents/`
  - `project-code/`
  - `common/`
- 组件只负责局部 UI 和局部交互，不直接承担跨页面编排。

### `src/api/`

- 放统一的数据模型、接口类型和请求函数。
- 页面和组件不要直接写 `fetch` / `axios`。
- 一个域一份 API 文件，保持边界清晰。

## 3. 标准页面壳

### 共享壳

- 统一使用 `WorkspacePanel` 表达“标题 + 操作区 + 内容区”。
- 适合：
  - 详情页
  - 列表页
  - 左右分栏工作台
  - 带顶部动作的编辑器页

### 双栏页面

- 左侧：列表 / 树 / 导航。
- 右侧：详情 / 编辑器 / 预览。
- 页面只负责左右栏数据同步，不在页面里重复写两套壳。

### 消息页

- 左栏：会话列表。
- 中间：消息流 + 输入框。
- 右栏：聊天管理 / 任务规划 / 文件目录，按需出现。
- 输入、@、发送、侧栏开关都交给组件处理。

## 4. Element Plus 使用优先级

### 优先使用

- 列表：`el-table`
- 详情：`el-descriptions`
- 容器：`el-card`
- 弹层：`el-dialog`
- 侧栏：`el-drawer`
- 空态：`el-empty`
- 提示：`el-alert`
- 表单：`el-input`、`el-select`、`el-switch`、`el-checkbox`、`el-input-number`

### 尽量少写的内容

- 自绘按钮组
- 自绘卡片栅格
- 自绘字段块
- 自己实现的简单列表项
- 重复的标题栏和操作栏

### 可接受的自定义

- 只做外层间距、圆角、背景、边框和布局比例调整。
- 不要为了“统一视觉”而重复造表格、弹窗、抽屉、列表。

## 5. 常见结构模板

### 列表页

- 顶部：标题 + 搜索 + 刷新 + 创建
- 中部：`el-table`
- 底部：分页或空态

### 详情页

- 顶部：标题 + 返回 + 刷新 + 保存
- 左侧：文件树 / 列表
- 右侧：详情 / 编辑器 / 预览

### 侧栏页

- 顶部：标题 + 关闭按钮
- 中部：分块卡片或表格
- 底部：错误提示 / 操作按钮

### 编辑器页

- 左侧：文件树 / 目录 / 列表
- 右侧：编辑器 + 工具区
- 编辑器内外层分离：
  - 外层负责布局和动作
  - 内层负责内容编辑本身

## 6. 样式约定

- 优先使用统一命名：
  - `shell`
  - `panel`
  - `panelHeader`
  - `panelBody`
  - `actions`
  - `toolbar`
  - `grid`
  - `list`
  - `detail`
- 只保留少量局部样式，避免每个页面写一大段重复 CSS。
- 同类页面尽量复用同一套间距、圆角、阴影、标题字号。
- 新增布局前先看 `WorkspacePanel` 和现有页面是否已经有现成结构可以复用。

## 7. 代码边界原则

- **页面负责**：路由、请求、状态、事件串联。
- **组件负责**：展示、局部交互、局部校验、局部开关。
- **API 负责**：类型、请求函数、接口参数。
- **不要把请求散到按钮里**，按钮只触发事件，具体请求放在页面或 API 层。

## 8. 新增页面时的默认做法

1. 先判断是否能复用 `WorkspacePanel`。
2. 再判断是否已有同类组件可复用。
3. 先用 Element Plus 现成组件拼结构。
4. 只有当现成组件覆盖不了时，才写少量自定义 UI。
5. 页面里不要直接堆复杂 HTML，优先拆成独立组件。

## 9. 维护建议

- 新增或修改一类页面时，先把共性提炼成组件，再改页面。
- 如果某块结构在两个页面以上出现，优先抽到 `src/components/common/` 或对应业务目录。
- 如果某个样式需要复制三次以上，考虑抽成共享组件或共享样式规范。
