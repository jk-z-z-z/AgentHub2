# AgentHub 技术文档

## 1. 技术目标

AgentHub 的技术设计目标是构建一个支持多人多 Agent 协作的执行底座。系统需要同时满足以下要求：

- 支持单聊、项目群聊、bootstrap 初始化三种不同会话模式。
- 支持 manager / worker / bootstrap 三类 Agent 运行时解耦。
- 支持复杂任务的 `Plan and Execute` 模式与简单任务的直达执行模式。
- 通过事件驱动而非阻塞式调用，支持异步子任务执行。
- 将消息、事件、任务图、工具调用、代码变更、部署动作统一沉淀到一个项目上下文。

## 2. 多 Agent 协作与事件驱动总览

### 2.1 为什么这部分要前置

对于 AgentHub 而言，最核心的技术问题是让多个 Agent 在团队场景中稳定协作。因此文档优先解释协作模型和事件驱动机制，再展开模块划分与实现细节。

### 2.2 协作模型

系统当前支持三类核心参与者：

- `user`：提出目标、补充约束、查看过程和确认结果
- `manager agent`：负责规划、调度、复核和协作收口
- `worker agent`：负责执行具体任务、工具调用和产物生成

协作路径分为两类：

- `Plan and Execute`：复杂任务由 manager 拆成 DAG，再分派给 worker
- `Direct Execute`：简单任务由用户直接 `@worker agent`

### 2.3 事件驱动总流程

系统的关键流程围绕事件链路展开：

1. 用户消息进入系统
2. 系统创建 `message.created` 事件
3. dispatcher 根据群聊类型和上下文选择 manager / worker / bootstrap runtime
4. manager 或 worker 在执行中写入过程事件
5. 若是节点任务，manager 发出执行请求事件
6. worker 异步消费事件并执行
7. worker 写回 `task.completed` 或 `task.failed`
8. manager 再次收到结果事件并执行复核

### 2.4 事件驱动带来的好处

- 主流程不阻塞等待子 Agent
- 支持并发子任务
- 执行过程可审计、可追踪
- 更适合失败重试、重入队和任务恢复
- 更符合多人多 Agent 协作场景

## 3. 总体架构

### 3.1 技术栈

- 后端：`FastAPI`
- 数据层：`SQLAlchemy + SQLite`
- Agent 运行基础：`AgentScope`
- 前端：`Vue 3 + Element Plus + Vite`

### 3.2 后端模块划分

基于当前代码，后端主要由以下模块组成：

- `api/`：提供会话、成员、消息、任务、Agent、工具、部署等接口。
- `services/`：提供群聊、任务节点、项目代码、工作区、部署等领域服务。
- `event_runtime/`：统一事件记录、上下文拼接与事件分发。
- `agent_runtime/`：数字员工执行运行时。
- `manager_runtime/`：管家 Agent 执行运行时。
- `bootstrap_runtime/`：Agent 初始化运行时。
- `memory_runtime/`：记忆配置、压缩与 token 估算。
- `ws_runtime/`：消息和状态的 WebSocket 推送。

### 3.3 前端模块划分

前端以 `MessagesPage.vue` 为协作中心，核心能力包括：

- 会话列表与会话切换
- 消息线程与消息发送
- Agent mention
- 项目代码侧栏
- 部署与预览侧栏
- 任务规划侧栏（Run / DAG / 节点）
- 聊天管理与成员管理

## 4. 核心领域模型

### 4.1 会话与成员

- `Group`：会话容器，当前支持 `personal`、`project`、`bootstrap`。
- `Member`：会话成员，支持 `user`、`agent`、`system`。
- `GroupAssistantConfig`：项目群聊的管家配置，决定是否启用管家 Agent。

### 4.2 消息与事件

- `Message`：消息展示实体。
- `MessageEvent`：消息关联的事件链，是系统真实的过程记录来源。

消息负责展示，事件负责驱动与审计，这是当前实现中的一个重要设计原则。

### 4.3 任务编排

- `GroupTaskRun`：一次任务规划运行。
- `GroupTaskNode`：Run 中的 DAG 节点。

一个项目群聊下可以有多个 Run；每个 Run 包含一张可持续编辑的 DAG。

### 4.4 执行与部署

- `Workspace`：项目工作区。
- `ExecutionJob` / `SandboxRun`：命令执行与沙箱任务。
- `PreviewJob`：预览任务。
- `DeploymentJob`：部署任务。

## 5. 消息与事件驱动架构

### 5.1 入口链路

消息写入入口为 `message_service.create_message_and_trigger_ai(...)`，写入消息后会异步触发 `message.created` 事件。后续由 `event_runtime.dispatcher` 按消息维度顺序消费 pending 事件。

主链路如下：

`create_message_and_trigger_ai`
-> `agent_runtime.message_store.create_message`
-> `message.created`
-> `event_runtime.handlers.message/task/...`
-> `bootstrap_runtime / agent_runtime / manager_runtime`

### 5.2 事件分类

根据当前实现，事件分为：

- `input_output`：消息输入、回复开始、回复完成、失败等
- `dag`：DAG 编辑、节点变更
- `execution`：思考、tool call、tool result、运行过程
- `task`：节点执行请求、完成、失败、复核、重入队
- `system`：bootstrap、系统兜底类事件

### 5.3 事件驱动价值

事件驱动是本系统支持多 Agent 协作的基础：

- 主 Agent 不需要同步等待子 Agent 完成。
- 子 Agent 的执行痕迹可以完整记录。
- 管家可以在异步结果返回后再进行复核。
- 失败重试、重入队、审计追踪都更容易实现。

## 6. 三类 Agent 运行时设计

### 6.1 Manager Agent

`manager_runtime` 负责项目群聊中的管家能力，包括：

- 澄清与收敛用户目标
- 规划任务与维护 DAG
- 节点分配与执行请求发起
- 节点完成后的结果复核
- 必要时补充调整 DAG、返工或重新排队

Manager 是群聊层的系统治理能力。

### 6.2 Worker Agent

`agent_runtime` 负责普通数字员工能力，包括：

- 单聊自动回复
- 项目群聊中的被提及执行
- DAG 节点执行
- 工具调用、MCP 调用、技能加载

Worker Agent 面向“执行”，不承担项目级规划与复核职责。

### 6.3 Bootstrap Agent

`bootstrap_runtime` 负责 Agent 正式进入协作前的初始化流程，包括：

- 读取 `SOUL.md`、`PROFILE.md`、`BOOTSTRAP.md`
- 加载 bootstrap 专属 skill 与工具
- 初始化工作区、资料与基础角色信息

Bootstrap 与正式执行隔离，避免初始化逻辑污染项目协作逻辑。

## 7. 上下文拼接策略

### 7.1 统一原则

系统当前遵循“外部负责传入短期上下文和运行时上下文，运行时内部负责拼装角色资源”的边界：

- `short_term_memory`：由事件流拼接。
- `extra_context` / `runtime_context`：由外部链路注入，如 `group_id`、`project_id`、`user_id`、`run_id`、`node_id`、`input_text`。
- Agent 固有材料：由各 runtime 内部从文件系统读取，如 `SOUL.md`、`PROFILE.md`、`MEMORY.md`。

### 7.2 单聊上下文

单聊场景下，worker agent 的 system prompt 主要由以下部分组成：

- Agent 的 `SOUL.md`
- Agent 的 `PROFILE.md`
- User 的 `PROFILE.md`
- User 的 `MEMORY.md`
- 固定角色说明

短期上下文来自当前会话的事件流摘要。

### 7.3 项目群聊上下文

项目群聊场景下，worker agent 的 system prompt 主要由以下部分组成：

- Agent 的 `SOUL.md`
- Agent 的 `PROFILE.md`
- Project 的 `PROFILE.md`
- Project 的 `MEMORY.md`
- 固定角色说明

Manager agent 的上下文则更偏治理视角，除短期摘要外还会引入：

- 项目根目录
- 项目 `README.md` 摘要
- 项目 `MEMORY.md` 摘要
- `knowledge/` 文档预览
- 当前群聊可分配 Agent 列表
- 对话短期摘要
- Manager skill prompt sections

### 7.4 Bootstrap 上下文

bootstrap system prompt 主要由以下内容拼接：

- Agent 的 `SOUL.md`
- Agent 的 `PROFILE.md`
- Agent 的 `BOOTSTRAP.md`
- User 的 `PROFILE.md`
- User 的 `MEMORY.md`
- bootstrap 角色说明

## 8. 任务执行模式

### 8.1 Plan and Execute

复杂任务走管家模式：

1. 用户在项目群聊中 `@管家`
2. manager_runtime 结合项目上下文生成规划
3. 规划结果落为 `GroupTaskRun + GroupTaskNode`
4. 管家通过 `manager.node_execute` 等工具发起节点执行
5. 子 Agent 通过事件异步执行节点
6. 管家根据 `task.completed / task.failed` 进行复核

### 8.2 Direct Execute

简单任务走直达模式：

1. 用户在项目群聊中直接 `@子 Agent`
2. 系统创建占位 AI 消息
3. worker agent 直接执行并回复
4. 若消息带有任务上下文，后续仍可触发管家评估

### 8.3 节点完成判定

从当前 `task.py` 的实现看，节点执行结果会结合真实产物进行判定，如：

- 代码写入
- 命令验证
- 预览结果
- 部署结果

若未检测到有效产物，节点可能被保持为 `pending`，避免虚假完成。

## 9. Tool / Skill / MCP 机制

### 9.1 Worker Agent

worker agent 的 toolkit 由以下来源构成：

- builtin tools
- agent 级启用的工具配置
- skill 池加载的 skills
- MCP / ACP 扩展能力

### 9.2 Manager Agent

manager agent 使用专门的工具集，重点围绕任务编排与节点管理，包括：

- DAG 查看与修改
- 节点认领、指派、完成、重入队
- 节点执行触发
- 记忆压缩等项目治理能力

### 9.3 Bootstrap Agent

bootstrap agent 只加载初始化所需的技能和工具，不参与项目期任务执行。

## 10. 前端交互与后端能力映射

结合当前前端实现，可将能力映射为：

- 会话列表：展示 `personal / project / bootstrap`
- 聊天主区：展示消息流与 mention 输入
- 项目群聊右侧栏：代码、部署、任务规划、聊天管理
- 任务规划面板：Run 列表、DAG 图、节点列表、认领/完成/复核

这说明产品交互已围绕“聊天驱动协作 + 任务图可视化治理”建立基本闭环。
