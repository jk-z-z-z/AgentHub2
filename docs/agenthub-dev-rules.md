# AgentHub 开发经验 Rules

## 1. 目的

本文档记录在开发 AgentHub 过程中沉淀出的 AI 协作开发规则。这些规则用于约束开发节奏、任务分配方式、代码组织方式和复查方式，帮助 AI 在复杂项目中保持更稳定的输出质量。

## 2. 任务规则

### Rule 1：先拆模块，再交任务

在把任务交给 AI 之前，先拆出模块边界，再定义本轮目标。

### Rule 2：每轮只解决一个小目标

每次任务应尽量只覆盖一个局部目标，避免把过大的需求一次性交给 AI。

### Rule 3：任务必须可验证

每一轮任务都应能在当前轮次中判断结果是否正确，避免依赖大范围联调后才知道是否偏航。

## 3. 实现规则

### Rule 4：第一轮优先完成功能

功能跑通优先于首轮结构完美。第一轮实现应优先确保需求被落地。

### Rule 5：第二轮必须安排重构

模块功能稳定后，应安排单独一轮结构性重构，提升代码可读性、职责边界和可维护性。

### Rule 6：不要长期保留大单文件

单文件集中实现可以用于原型或第一轮开发，但不应长期保留为最终结构。

## 4. 代码组织规则

### Rule 7：目录结构必须可导航

项目结构应让人和 AI 都能先读目录，再按需进入文件。

### Rule 8：文件职责必须清晰

每个文件应尽量承担明确职责，避免在同一文件中持续混入过多不同层次的逻辑。

### Rule 9：代码结构要降低后续 token 成本

代码组织应服务于后续 AI 二次开发，减少必须全量加载的大文件和高耦合结构。

## 5. 评审规则

### Rule 10：方案完成后必须做批判性复查

一个方案被设计出来后，应再进行一轮独立评估，检查是否存在风险、复杂度过高、边界遗漏或更优解。

### Rule 11：代码完成后必须做批判性复查

代码完成后，不只看“能不能跑”，还要看职责划分、可维护性、可扩展性和后续接手成本。

### Rule 12：不要默认 AI 会主动反驳当前思路

AI 往往会顺着当前思路继续推进，因此需要主动要求它从批判性角度重新审视方案和实现。

## 6. 协作规则

### Rule 13：一次生成效果不代表长期效果

首轮生成顺畅不等于结构合理，长期可维护性优先于一次性生成体验。

### Rule 14：开发流程要兼顾人和 AI 的接手成本

代码结构、任务描述和重构节奏都应兼顾后续人类开发者和 AI 的继续开发成本。

### Rule 15：复杂项目优先使用方法，而不是依赖单次提示词

当项目变复杂时，真正决定质量的是任务拆分、结构治理和复查机制，而不是某一次 prompt 写得多详细。

## 7. 总结

这些规则的核心价值在于：让 AI 在复杂项目中做得更稳，而不只是做得更快。它们共同服务于一个目标，即降低偏航、降低返工、降低后续理解成本。
