# AgentHub 产品文档

## 1. 产品概述

### 1.1 产品定位

AgentHub 是一个面向团队协作场景的多人多 Agent 协作系统，核心目标是让一个团队在使用 AI 进行研发协作时，始终保持目标一致、上下文一致、交付结果一致。

产品当前以会话为主入口，提供两类核心协作模式：

- 单人聊天：用户与自己的数字员工协作，完成个人任务、需求整理、代码修改、资料沉淀等工作。
- 群聊协作：多个用户与多个 Agent 在同一项目上下文中协作，由管家 Agent 负责任务理解、拆解、分配、复核和收口。

### 1.2 要解决的问题

在传统 AI Coding 场景中，常见模式是“一个人对应一个 Agent”或“一个人同时驱动多个 Agent”。这类模式在个人效率上有效，但在团队协作中会暴露出明显问题：

- 团队成员各自与 AI 对话，需求理解容易分叉。
- 多个开发者分别让 AI 生成代码，最终产品方向可能偏移。
- 任务拆解、执行与验收分散，缺少统一的过程治理。
- Agent 之间没有共享协作协议，复杂任务难以并行推进。
- 主控 Agent 若必须等待子 Agent 完成，会导致长链路阻塞，交互体验差。

AgentHub 的目标是通过共享上下文、统一任务编排、事件驱动执行与管家复核机制，降低团队在 AI 参与研发后的“目标漂移”和“协作失控”问题。

## 2. 核心协作理念

### 2.1 协作是产品核心

AgentHub 以聊天作为产品入口，以多人多 Agent 协作作为核心能力。

项目重点聚焦在以下能力：

- 多个用户如何围绕同一目标协作
- 多个 Agent 如何在同一项目上下文中分工
- 复杂任务如何被规划、拆解、执行、复核
- AI 参与研发后，如何避免目标偏移和结果失控

因此在产品叙事上，聊天界面承载协作入口，协作机制承载核心产品价值。

### 2.2 多人多 Agent 协作方式

系统当前主要支持两类协作方式：

- `单聊协作`：用户与个人数字员工协作，强调即时执行和个人效率。
- `群聊协作`：多个用户、多个 Agent 在同一项目会话中协作，强调共享上下文、任务分工和统一收口。

在群聊场景中，协作进一步拆分为两条路径：

- `复杂任务协作`：由管家 Agent 规划任务、生成 DAG、分配子 Agent 执行。
- `简单任务协作`：用户直接 `@子 Agent` 执行具体事项。

### 2.3 为什么采用事件驱动

多人多 Agent 协作的关键约束在于，子 Agent 的执行往往不是瞬时完成的。如果主 Agent 必须同步等待子 Agent 返回，系统会出现以下问题：

- 群聊主流程被阻塞，用户等待时间长
- 管家无法继续调度其他任务
- 并行子任务难以推进
- 执行过程难以审计和回放

因此 AgentHub 采用事件驱动机制：

- 管家发送执行事件
- 子 Agent 在后台异步执行节点
- 执行结果通过完成/失败事件回流
- 管家收到事件后再复核、补充调度或向用户汇报

这使系统从“对话式 AI”升级为“可并行、可追踪、可治理的协作系统”。

## 3. 用户与场景

### 3.1 核心用户

- 产品经理：在群聊中提出需求、澄清目标、查看规划结果与交付状态。
- 开发工程师：与个人 Agent 或项目 Agent 协作完成编码、调试、验证和部署。
- 技术负责人：通过管家 Agent 查看 DAG、任务分工、风险节点和执行结果。
- AI Agent 设计者：配置 Agent 角色、工具、技能和工作区初始化内容。

### 3.2 核心使用场景

#### 场景 A：个人高频执行

用户在单聊中直接与自己的数字员工沟通，完成快速编码、文档编写、资料整理、问题排查等任务。

产品价值：

- 提供低门槛的个人 AI 助手体验。
- 保留用户画像、记忆和个人上下文。
- 为后续进入项目协作提供统一 Agent 资产基础。

#### 场景 B：项目群聊协作

用户在项目群聊中与其他成员和多个 Agent 协作。复杂需求由管家 Agent 进行规划和拆解，生成 DAG；简单任务则可直接 `@指定子 Agent` 执行。

产品价值：

- 让团队围绕同一个项目目标协作。
- 把需求讨论、任务分解、执行过程和复核结果放在一个会话空间中。
- 降低成员分别使用 AI 导致的结果偏移。

#### 场景 C：Agent 初始化与入场

新建 Agent 后，系统为其创建 bootstrap 会话，由 bootstrap agent 完成工作区初始化、资料准备和角色设定。

产品价值：

- 让 Agent 在正式参与协作前具备最小可用上下文。
- 将初始化和正式执行解耦，降低角色混乱。

## 4. 产品目标

### 4.1 业务目标

- 建立团队共享的 AI 协作工作台。
- 支持复杂研发任务的多人多 Agent 协同。
- 把 AI 生成能力升级为“可管理、可追踪、可复核”的协作流程。

### 4.2 体验目标

- 用户只需在聊天界面中发起任务，即可完成协作编排。
- 复杂任务有清晰的任务图、责任人和状态反馈。
- 子 Agent 执行不阻塞主流程，用户能持续获得过程反馈。

### 4.3 成功指标建议

- 项目群聊中由管家生成 DAG 的任务占比。
- DAG 节点自动分配成功率。
- 任务从提出到首次可执行规划的平均耗时。
- 子 Agent 节点一次通过率。
- 团队协作中因需求理解偏差导致的返工率。
- 单聊与群聊的周活跃用户数、会话数、任务完成数。

## 5. 产品设计

### 5.1 会话模型

系统当前围绕三种会话容器展开：

- `personal`：单聊，仅支持用户与一个 Agent 协作，不支持 `@` 提及。
- `project`：项目群聊，支持多个用户、多个 Agent 和一个可选的管家 Agent。
- `bootstrap`：初始化会话，用于 Agent 入场前的 bootstrap 流程，不支持 `@` 提及。

### 5.2 群聊协作模型

项目群聊是产品核心，具备以下能力：

- 成员管理：支持用户成员与 Agent 成员共同加入同一会话。
- 管家开关：群聊可启用或关闭管家 Agent。
- 任务规划：支持创建 Run、查看 DAG、查看节点列表与状态。
- 任务执行：节点可认领、分配、执行、完成、复核、返工。
- 项目资产：群聊与共享代码空间、部署、预览、长期记忆关联。

### 5.3 单聊协作模型

单聊强调即时性和个人效率：

- 用户发送消息后自动触发该会话内唯一 Agent 回复。
- 依赖用户画像、用户记忆和 Agent 自身画像形成上下文。
- 适合个人需求澄清、辅助写码、沉淀个人偏好。

### 5.4 任务处理模式

产品同时支持两种任务路径：

- 复杂任务：由管家 Agent 执行 `Plan and Execute`，先规划后分发。
- 简单任务：用户在群聊中直接 `@子 Agent`，跳过规划直接执行。

这种双路径设计兼顾了复杂协作治理与轻量即时执行。

## 6. 核心流程

### 6.1 单聊流程

1. 用户进入单聊并发送文本消息。
2. 系统从事件流拼接短期上下文。
3. 自动找到该会话中的唯一 Agent。
4. Agent 基于用户上下文和个人记忆执行回复或工具调用。
5. 结果回写消息与事件流。

### 6.2 群聊复杂任务流程

1. 用户在项目群聊中 `@管家` 提出需求。
2. 管家读取项目资料、对话摘要、可分配 Agent 列表等上下文。
3. 管家生成或更新任务 Run 与 DAG。
4. 节点按角色要求分配给子 Agent 或成员。
5. 管家通过事件发起节点执行。
6. 子 Agent 异步处理节点，输出结果与执行痕迹。
7. 管家接收完成/失败事件并进行复核。
8. 必要时更新 DAG、重试节点或向用户汇报阶段结果。

### 6.3 群聊简单任务流程

1. 用户在项目群聊中直接 `@某个 Agent`。
2. 系统为被提及 Agent 创建执行占位消息。
3. Agent 基于项目上下文直接执行任务。
4. 结果写回群聊；若带有任务上下文，则可再触发管家评估。

## 7. 差异化价值

与“单人单 Agent”或“单人多 Agent”方案相比，AgentHub 的差异主要在于：

- `共享上下文`：协作围绕同一个项目会话、项目记忆与共享代码空间展开。
- `统一编排`：复杂任务交由管家生成 DAG，而非成员各自拆解。
- `多角色协同`：用户、管家、子 Agent 各有明确职责边界。
- `事件驱动`：主流程不阻塞等待子 Agent，适合长任务和并行任务。
- `过程可见`：消息、事件、任务节点、复核状态都可追踪。

## 8. 当前能力边界

结合现有代码实现，当前版本建议明确以下边界：

- 单聊和 bootstrap 会话均不支持 Agent mention。
- 单聊当前仅适合“一个用户 + 一个 Agent”的轻协作模式。
- 项目群聊当前只支持一个全局管家 Agent 配置。
- DAG 以单个群聊内的 Run 为单位管理，当前聚焦群聊内任务规划与执行。
- 节点执行结果是否完成，当前会参考是否产生真实代码写入或验证产物。

## 9. 后续演进建议

- 引入跨会话的产品需求对象与版本对象，减少需求只沉淀在聊天里的问题。
- 为群聊增加更清晰的需求确认、验收标准和变更记录机制。
- 增加更多自动分派策略，如按角色、技能标签、历史表现分配 Agent。
- 提升事件流压缩与长期记忆能力，降低项目上下文膨胀。
- 建立项目级别的“目标对齐面板”，将需求、设计、DAG 和交付结果统一展示。
